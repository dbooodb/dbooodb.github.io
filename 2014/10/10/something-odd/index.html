Oracle 10g在安全加固环境安装的问题

  • Oracle

对于DB圈子多年的O记老司机，install and uninstal这样的事情本该是三下五除二很快搞定的。

是的，在一个完全没有限制的干净环境剧情应该是这样走的。

最近接到一个在安全加固系统安装10g的问题。

这是一个测试环境，安装10g用于测试升级至11g，客户基于安全考虑，提供的环境做过一系列安全增强的修改，这使得事情没那么顺利。

  • OS version： Redhat 5.4
  • DB version： Oracle 10.2.0.5

Issue 1: 节点信任

完成节点对等配置之后，测试对等情况，反应有点缓慢，报了两个waring之后对等通过。

Warning: untrusted X11 forwarding setup failed: xauth key data not generated
Warning: No xauth data; using fake authentication data for X11 forwarding.

本以为这样没有问题那就错了，后面安装database环节会检测到节点不对等，安装失败。

咨询主机相关技术人员，答复确实对ssh做了修改，但至于为什么导致响应缓慢，他们也不清楚，结论就是要另寻他法自性解决。

查阅很久有关这两个waring的问题，没有找到解决办法，最后只能放弃ssh，改用rsh配置节点对等。

# chkconfig rsh on
# chkconfig rlogin on
# service xinetd reload

以root用户创建/etc/hosts.equiv文件

touch /etc/hosts.equiv
chmod 600 /etc/hosts.equiv
chown root:root /etc/hosts.equiv

编辑/etc/hosts.equiv文件，把两个节点加入进去

+TEST-SOADB01 oracle
+TEST-SOADB02 oracle
+TEST-SOADB01-PRIV oracle
+TEST-SOADB02-PRIV oracle

验证双机rsh互信情况：
rsh TEST-SOADB01 date
rsh TEST-SOADB02 date
rsh TEST-SOADB01-PRIV date
rsh TEST-SOADB02-PRIV date

Issue 2: 节点不可达

节点信任问题解决之后，走到安装环境又报了另外一个错误，节点不可达，再一次异常退出，这就尴尬了。

在shell下用ping检查节点可达性，没有问题。

安装程序会自行调用部分操作系统命令做检查，推测可能是权限设置导致。

检查了一下ping命令的权限，果然做了手脚，修改回来：

# chmod 4755 /bin/ping    
 修改后的权限信息应该是这样: 
 -rwsr-xr-x 1 root root 37312 Apr 24  2009 /bin/ping

Issue 3: 无法识别cluster

执行orainstRoot.sh, root.sh，出现无法识别cluster，原因是Oracle无法读取/etc/oraInst.loc文件，安全加固真特么到底把系统改成了什么乱七八糟的。

解决办法：

以root用户执行root.sh前先检查umask情况：
# umask
0027
修改为0022，命令：
# umask 022
# umask
0022
然后下面可以开始当前的root session去执行脚本了。（特别注意：重新登录root之后
umask还是会变成027，需再次修改为022）

END

后面的安装就没有什么问题了。

注重安全是好，但需要事先做好影响度评估，不能为了加固而加固。

这次遇到的只是安装问题，也就麻烦一点罢了。

生产环境已经遇到过N次因为安全加固把生产数据库搞宕机的问题，引以为戒。
